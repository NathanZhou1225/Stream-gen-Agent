## ⚠️ 二、API 消耗高的根本原因分析

### 【问题 1】Schema 验证的「瀑布式试错」成本极高

**现象**: draft_manager 的 schema 验证每次**只返回第一个遇到的错误**，Agent 必须逐个修复后重试，形成「错误瀑布」：

```
第 1 次: SCRIPT_APPENDIX_REQUIRED → 加 production_appendix
第 2 次: SCRIPT_APPENDIX_SECTION_REQUIRED (camera_shots) → 加 camera_shots 数组
第 3 次: SCRIPT_APPENDIX_SECTION_REQUIRED (stickers_effects) → 加 stickers_effects 数组
第 4 次: SCRIPT_APPENDIX_SECTION_REQUIRED (visual_assets) → 加 visual_assets 数组
第 5 次: SCRIPT_APPENDIX_SECTION_REQUIRED (host_actions) → 加 host_actions 数组
第 6 次: SCRIPT_SEGMENTS_MISSING → 把 script 改成 segments
第 7 次: SCRIPT_SCHEMA_MISSING_FIELD (draft_id) → 加 draft_id
第 8 次: SCRIPT_SCHEMA_MISSING_FIELD (duration_sec) → 加 duration_sec
第 9 次: SCRIPT_SCHEMA_SEGMENT_INVALID (time) → 把 timestamp 改成 time
第 10 次: SCRIPT_SCHEMA_SEGMENT_INVALID (role) → 加 role 字段
第 11 次: SCRIPT_SCHEMA_SEGMENT_INVALID (say) → 把 content 改成 say
第 12 次: 终于通过！
```

**影响**: 仅 script_refining 阶段就消耗了 **10+ 次 exec**，占总消耗的 30% 以上。

### 【问题 2】Schema 要求分散且不透明

**现象**: Schema 要求分散在 draft_manager.py 的多个 `_assert_*` 函数中，没有统一文档：

- ✅ `_assert_outline_schema()` - 要求每个 point 有 `production_hint`（≤36 字）
- ✅ `_assert_script_appendix_schema()` - 要求 `production_appendix` 包含 4 个固定数组（每数组 3-5 条）
- ✅ `_assert_script_fact_opinion_schema()` - 特定 role 必须有 `claim_kind` + 证据来源
- ✅ `_assert_script_style_adaptation()` - 有 user_style_context 时必须加风格适配
- ✅ `render_script_md()` - 要求顶层 `draft_id` + `duration_sec` + `segments[]`

**影响**: Agent 只能通过报错反推 schema 要求，无法一次性构造正确 payload。

### 【问题 3】字段命名不一致，容易混淆

| 错误命名 | 正确命名 | 发现方式 |
|----------|----------|----------|
| `script[]` | `segments[]` | SCRIPT_SEGMENTS_MISSING |
| `timestamp` | `time` | SCRIPT_SCHEMA_SEGMENT_INVALID |
| `content` | `say` | SCRIPT_SCHEMA_SEGMENT_INVALID |

### 【问题 4】可选校验项触发条件不明确

- **role 校验**: 只有当 role 在 `{'argument_1', 'argument_2', 'argument_3', 'argument', 'turn', 'scene', 'conflict', 'result', 'action'}` 中时，才需要 `claim_kind` + 证据来源。使用 `host`、`opening` 等则跳过。
- **风格适配校验**: 只有当 payload 含 `user_style_context` 时，才需要 `production_style_adaptation`。

**影响**: 不了解这些「豁免条件」时，会过度构造 payload 导致额外消耗。

## 🚀 四、优化建议（立即生效可减少 50%+ 消耗）

### 【短期优化】（可立即实施）

1. **把上述完整 Schema 写入 Memory 或 Skill 提示词**
   - 让 Agent 一次性构造正确 payload，无需「瀑布式试错」
   - 预估节省：**8-10 次 exec/轮**

2. **使用 `role: "host"` 而非 `role: "argument_1"` 等**
   - 跳过复杂的事实/观点标注校验
   - 预估节省：**3-5 次 exec/轮**

3. **无特殊需求时不传 `user_style_context`**
   - 跳过风格适配校验
   - 预估节省：**2-3 次 exec/轮**

### 【中期优化】（工具侧改进）

1. **draft_manager 一次性返回所有 schema 错误**（而非只返回第一个）
   - `errors: [{code, message}, ...]` 数组形式
   - 一次性修复所有问题

2. **提供 `--example-schema outline|script` 子命令**
   - 直接输出可通过校验的最小 JSON 示例
   - Agent 只需填空无需猜结构

3. **提供 `doctor --schema outline|script` 诊断**
   - 提前列出所有要求字段和可选字段

### 【长期优化】（架构层面）

1. **payload 构造函数化**
   - 提供 Python 辅助函数 `make_outline_payload()` / `make_script_payload()`
   - 返回结构化正确的 dict，Agent 只需填内容

2. **错误代码可搜索**
   - 每个错误码对应在线文档链接，包含修复示例
