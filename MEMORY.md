### 4.1 行情 / 热点 / 信源快照（与 `finance-source-ingest` 对齐）

1. **唯一信源形态**：在 workspace 下执行 **`python3 skills/finance-source-ingest/scripts/ingest.py run --sources market,news,social`**（参数以 `natural-language-intent.md` §4.4 为准）。
2. **展示契约**：将 stdout JSON 里的 **`markdown_summary` 全文原样**发给用户（可加一行采集时间）；**禁止**只贴三大指数表、**禁止**只贴财联社/板块表、**禁止**按用户口头词只跑 `market` 或 `news`、**禁止**把「行情」和「热点」拆成两条消息或两套结构。
3. **结构说明**：`markdown_summary` 已内建 **大盘与情绪（三大指数优先 → 北向资金 → 其他情绪/资金）→ 六大核心板块快讯（每条带时间戳；无最新命中时可用关键词回溯补位；新闻池仍无命中时用行情侧补充或明确暂无）→ 大事件（国家/全球/政策/地缘/峰会类）→ 金融相关今日热点 → 社媒/人气榜探测 → 中文告警**；百度热榜只展示有财经详情的条目，不再回退展示非财经标题。不要再用本文件旧版「五段式」去覆盖或删减这些块。
4. **旧版五段式**（大盘→焦点→新闻摘要→海外→选题）**仅**保留给：用户明确要求「写一份你手搓的盘面解读/复盘」、或对标账号分析/复盘技能等 **非 ingest 快照** 场景；与 §4.1 本条 **不**混用。

### 4.1A Agent WebSearch 兜底（只追加，不覆盖）

当 `markdown_summary` 已原样展示，但仍存在**板块无财联社正文、北向资金为空或 0、百度/泛财经无详情、社媒/人气榜/舆情为空、今日热点过薄**等缺口时，若返回 JSON 有 `meta.websearch_required: true` 或 `meta.websearch_gaps` 非空，具备 WebSearch 能力的 Agent **必须实际调用 WebSearch**，再在原文后追加 **「联网补充（Agent WebSearch 兜底）」**：

- **定位**：这是 Agent 层补充，不属于 `finance-source-ingest` 原始 API 快照；不得与财联社/百度/新浪 API 内容混排。
- **触发**：`meta.websearch_gaps` 非空；六大板块任一板块仅有「行情侧补充/暂无」；北向资金为空或 0；泛财经热榜无可展示财经详情；社媒/人气榜/舆情无条目；用户要求「更全」或「补一下缺失板块」。
- **格式**：每条必须写明 **归属板块/泛财经 + 摘要 + 来源 + 时间**；无网页发布时间时标注「检索时间」。
- **失败说明**：若由 API 失败触发，附录必须写明「因 XX 接口失败/为空，使用 WebSearch 兜底」；告警面向用户必须用中文，不直接裸露英文 traceback。
- **执行要求**：禁止只把「已触发/需 WebSearch」提示原样贴出就结束；必须给出 WebSearch 结果。若当前会话没有 WebSearch 工具或调用失败，必须追加「WebSearch 未执行成功：原因」，不得只写“建议联网搜索”。
- **工具来源**：WebSearch 可来自平台原生 WebSearch、Brave API，或当前 workspace 已安装且可执行的搜索 skill；不要把 Brave API 当成唯一可用方案。
- **Brave 未配置时的硬回退**：若调用了 `web_search` 且返回 `missing_brave_api_key` / 要求配置 Brave，**同轮**必须改用已安装的 **`tavily-search`**：`cd` 到 workspace 根后执行 `node skills/liang-tavily-search-1.0.1/scripts/search.mjs "<与 websearch_gaps 对应的查询>" -n 3 --json`（环境已有 `TAVILY_API_KEY`）；不得以「未配置联网搜索工具」代替执行。
- **大事件口径**：只收国家性/世界性、政策、地缘、战争冲突、峰会/协定等可能影响金融市场的事件；公司业绩、个股涨停、午评、IPO 辅导等只属于热点或板块，不放入大事件。
- **限流**：每个缺口板块最多 1 次搜索、1-2 条补充；总补充不超过 6-8 条。
- **红线**：WebSearch 不得覆盖 API 给出的指数、涨跌幅、北向资金等数值；只可补背景、事件或可核验新闻。搜不到就明确写「未找到可核验补充」，禁止脑补。

## 五、核心规则 - Workflow 流程规范

### 5.0 开稿前置流程（必须严格执行）
每次内容生成必须严格按照以下顺序执行，**不得跳步、不得遗漏**：
1. 第一步：若用户仅要盘面/信源事实，按 **§4.1** 跑 **全量 `ingest`** 并原样展示 `markdown_summary`；若用户已进入**开稿**链，则按 `streamy-content-gen` / `preflight` 展示 digest 与候选，**不要**用旧五段式替代 ingest 输出。
2. 第二步：主动询问用户是否开稿、朝哪个方向开稿
3. 第三步：用户确认开稿方向后，**必须先询问用户要使用的风格，且必须列出当前所有可用的风格名称供用户选择**
4. 第四步：确认风格后，再进入正式的「选题→大纲→逐字稿→定稿」三步骤流程

### 5.0A 明确方向优先（新增硬规则，优先级高于 5.0）
- 当用户首句或当前回合已明确给出**开稿方向**（例如“为我AI行情方面进行开稿”），**禁止**先走“通用热点兜圈子”再二次确认方向。
- 必须直接按该方向执行开稿链路：`create_draft -> preflight_topic.py --direction "<用户原话方向>" -> topic_picking`。
- 用户方向词必须原样进入 `topic_payload.direction`，且候选标题/论点必须与该方向语义相关；若相关性不足，必须当轮重生候选，不得直接展示。
- 用户表达「先看盘 / 只要事实 / 不开稿」时：**盘面事实**仍以 **§4.1 全量 `ingest` → `markdown_summary` 原样** 为准；**禁止**用旧五段式替代或删减 ingest 块。旧五段式仅用于 §4.1 第 4 点所列的**手写解读/复盘**场景。

### 5.1 稿件追踪与标识
- 在展示选题方向、成稿、状态更新等内容时，必须明确标注当前处理的草稿ID（如 **#QH3**），方便用户追踪和管理多篇稿件
- 稿件ID应在每次重要操作前醒目显示

### 5.2 成稿收尾流程
- 成稿完成后，必须主动询问用户两个问题：
  1. 是否确认定稿并归档此稿件？
  2. 如果确认归档，是否需要自动执行 auto-refine 来优化绑定的风格？
- 这是workflow收尾阶段的必要步骤，不得遗漏

### 5.3 异常处理机制
- 当执行 `--set-chosen` 出现 `TOPIC_CANDIDATES_EMPTY` 错误时，说明 `topic_candidates.json` 缺少 `candidates` 字段
- 处理方式：手动构建包含 `candidates` 数组的正确格式JSON文件，包含至少3个候选选题，并写入后重试

### 5.4 选题展示硬规则（飞书）
- 在 `topic_picking` 阶段，展示候选时**禁止**只给“选题标题 + 切入角度”
- 每个候选**必须同时展示**：`标题` + `核心论点(thesis)` + `3条论据(evidence)`
- 若本轮漏展示论点/论据，必须按同一 Draft 立即补发完整候选卡片
- 论据必须与用户方向同域（例如“AI行情”方向下，论据不可全部是无关宏观泛新闻）；若不相关，先重生候选再展示
